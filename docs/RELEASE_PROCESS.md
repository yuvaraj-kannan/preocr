# Release Process Guide

## Overview

PreOCR uses **automated CI/CD** for releases. The `ci-cd-release.yml` workflow handles everything automatically when you push to `main`.

## How It Works

### Automatic Version Bumping

The workflow uses **conventional commits** to automatically determine version bumps:

| Commit Type | Version Bump | Example |
|------------|--------------|---------|
| `fix:` | **Patch** (0.0.1 → 0.0.2) | `fix: resolve mypy errors` |
| `feat:` | **Minor** (0.1.0 → 0.2.0) | `feat: add new extraction feature` |
| `feat!:` or `BREAKING CHANGE:` | **Major** (1.0.0 → 2.0.0) | `feat!: change API signature` |

### Complete Release Flow

```
1. Make code changes
   ↓
2. Commit with conventional commit message
   ↓
3. Push to main (or merge PR)
   ↓
4. CI/CD automatically:
   - Analyzes commits since last tag
   - Determines version bump (patch/minor/major)
   - Updates version.py
   - Runs lint and tests
   - Creates git tag
   - Creates GitHub Release
   - Publishes to PyPI
```

## Step-by-Step Release Process

### Option 1: Automatic Release (Recommended)

**When to use:** Normal code changes (fixes, features)

1. **Make your code changes**
   ```bash
   # Edit files, fix bugs, add features
   ```

2. **Commit with conventional commit message**
   ```bash
   git add .
   git commit -m "fix: resolve mypy type errors"
   # OR
   git commit -m "feat: add new extraction feature"
   # OR
   git commit -m "feat!: breaking API change"
   ```

3. **Push to main (or create PR and merge)**
   ```bash
   git push origin main
   # OR create PR and merge
   ```

4. **CI/CD automatically:**
   - ✅ Bumps version (based on commit type)
   - ✅ Updates `version.py`
   - ✅ Runs lint and tests
   - ✅ Creates tag `vX.Y.Z`
   - ✅ Creates GitHub Release
   - ✅ Publishes to PyPI

**No manual steps needed!**

---

### Option 2: Manual Version Release

**When to use:** 
- You want to release a specific version (e.g., 1.0.1)
- You've already updated `version.py` manually
- You want to control the exact version

1. **Update CHANGELOG.md**
   ```bash
   # Add entry for new version
   ## [1.0.1] - 2026-02-06
   
   ### Fixed
   - Fixed mypy errors
   - Fixed lint issues
   ```

2. **Set version in `preocr/version.py`**
   ```python
   __version__ = "1.0.1"
   ```

3. **Generate release files** (optional but recommended)
   ```bash
   python scripts/generate_release.py v1.0.1
   ```

4. **Commit with "chore: bump version" message**
   ```bash
   git add .
   git commit -m "chore: bump version to 1.0.1"
   ```

5. **Push to main (or create PR and merge)**
   ```bash
   git push origin main
   ```

6. **CI/CD will automatically:**
   - ✅ Detect manual version bump (from commit message)
   - ✅ Run lint and tests
   - ✅ Create tag `v1.0.1` (if it doesn't exist)
   - ✅ Create GitHub Release
   - ✅ Publish to PyPI

**Note:** The workflow now automatically detects manual version bumps and publishes them!

## Commit Message Patterns

### Skip CI/CD Entirely
```bash
git commit -m "docs: update README [skip ci]"
# OR
git commit -m "chore: update docs [ci skip]"
```
**Effect:** Entire pipeline skipped (no lint, test, or publish)

### Skip Version Bump Only
```bash
git commit -m "refactor: improve code structure [no bump]"
# OR
git commit -m "test: add test cases [skip version]"
```
**Effect:** Lint and tests run, but no version bump or publish

### Manual Version Bump (Skip Auto-Bump)
```bash
git commit -m "chore: bump version to 1.0.1"
```
**Effect:** Skips auto-bump, uses version from `version.py`, publishes

## What Gets Created Automatically

### 1. Git Tag
- **Format:** `v1.0.1`
- **Message:** "Release v1.0.1"
- **Created by:** `ci-cd-release.yml` workflow

### 2. GitHub Release
- **Title:** "Release v1.0.1"
- **Description:** Auto-generated from commits or from `releases/v1.0.1.json` if available
- **Created by:** `ci-cd-release.yml` workflow

### 3. PyPI Package
- **Version:** Matches `version.py`
- **Published to:** https://pypi.org/project/preocr/
- **Created by:** `ci-cd-release.yml` workflow

### 4. Release Files (Optional)
- **Location:** `releases/v1.0.1.md` and `releases/v1.0.1.json`
- **Generated by:** `scripts/generate_release.py` (manual or CI/CD)
- **Used for:** GitHub Release descriptions, CI/CD queries

## Workflow Details

### `ci-cd-release.yml` Jobs

1. **version-bump**
   - Analyzes commits since last tag
   - Determines bump type (patch/minor/major)
   - Updates `version.py` (if needed)
   - Outputs: `should_bump`, `new_version`

2. **lint**
   - Runs Black (formatting)
   - Runs Ruff (linting)
   - Runs mypy (type checking)

3. **test**
   - Runs pytest across Python versions
   - Uploads coverage

4. **tag-and-publish**
   - Creates git tag
   - Creates GitHub Release
   - Builds package
   - Publishes to PyPI

5. **summary**
   - Creates workflow summary

## Best Practices

### 1. Always Use Conventional Commits
✅ **Good:**
```bash
git commit -m "fix: resolve type errors"
git commit -m "feat: add table extraction"
git commit -m "docs: update README"
```

❌ **Bad:**
```bash
git commit -m "fixed stuff"
git commit -m "update"
git commit -m "changes"
```

### 2. Update CHANGELOG.md Before Release
- Add entry for new version
- Include date: `## [1.0.1] - 2026-02-06`
- Use structured sections: `### Added`, `### Changed`, `### Fixed`

### 3. Test Before Pushing
```bash
# Run lint locally
python -m black preocr/ --check
python -m ruff check preocr/
python -m mypy preocr/ --ignore-missing-imports

# Run tests
pytest
```

### 4. Use PRs for Important Releases
- Create PR for review
- Merge to main triggers CI/CD
- Easier to catch issues before release

## Troubleshooting

### Version Not Bumping?

**Check:**
1. Are you using conventional commits? (`fix:`, `feat:`, etc.)
2. Is there a tag for the current version?
3. Are commits since last tag using conventional format?

**Solution:**
```bash
# Check last tag
git describe --tags --abbrev=0

# Check commits since last tag
git log $(git describe --tags --abbrev=0)..HEAD --oneline

# Ensure commits use conventional format
```

### CI/CD Not Publishing?

**Check:**
1. Did lint pass? Check GitHub Actions
2. Did tests pass? Check GitHub Actions
3. Is `should_bump == 'true'`? Check workflow logs

**Solution:**
```bash
# For manual release, ensure commit message is:
git commit -m "chore: bump version to X.Y.Z"

# Check workflow logs at:
# https://github.com/yuvaraj3855/preocr/actions
```

### Tag Created But No PyPI Release?

**Check:**
1. Is PyPI_API_TOKEN secret configured?
2. Did build step succeed?
3. Check workflow logs for errors

**Solution:**
- Verify PyPI_API_TOKEN in GitHub Secrets
- Check workflow logs for specific errors
